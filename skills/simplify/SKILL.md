# Simplify：代码审查与清理

审查所有已变更的文件，从复用性、质量和效率三个维度检查并修复发现的问题。

## 阶段一：识别变更

运行 `git diff`（如有暂存变更则使用 `git diff HEAD`）查看改动内容。如果没有 git 变更，则审查用户提到的或本次对话中编辑过的最近修改文件。

## 阶段二：并行启动三个审查代理

使用 Task 工具在一条消息中并行启动全部三个代理。将完整的 diff 传递给每个代理，确保它们拥有完整上下文。

### 代理 1：代码复用审查

对每处变更：

1. **搜索已有的工具函数和辅助方法**，寻找可替代新写代码的现有实现。在代码库中查找相似模式——常见位置包括工具目录、共享模块、以及变更文件的相邻文件。
2. **标记任何与已有功能重复的新函数。** 建议使用已有的函数替代。
3. **标记可以使用已有工具的内联逻辑** —— 手写字符串处理、手动路径处理、自定义环境检查、临时类型守卫等模式都是常见候选。

### 代理 2：代码质量审查

对同一批变更进行质量审查：

1. **冗余状态**：重复已有状态的状态、可以推导出的缓存值、可以改为直接调用的观察者/副作用
2. **参数膨胀**：向函数添加新参数而非泛化或重构现有参数
3. **复制粘贴微变体**：应通过共享抽象统一的近乎重复的代码块
4. **抽象泄漏**：暴露了应当封装的内部细节，或破坏了现有抽象边界
5. **字符串类型化**：在代码库中已有常量、枚举（字符串联合类型）或品牌类型的地方使用原始字符串
6. **不必要的 JSX 嵌套**：不增加布局价值的包装 Box/元素——检查内部组件的 props（flexShrink、alignItems 等）是否已提供所需行为

### 代理 3：效率审查

对同一批变更进行效率审查：

1. **不必要的工作**：冗余计算、重复文件读取、重复网络/API 调用、N+1 模式
2. **错失的并发机会**：可以并行运行的独立操作被串行执行
3. **热路径膨胀**：在启动或每请求/每渲染的热路径中添加了新的阻塞工作
4. **循环无效更新**：轮询循环、定时器或事件处理器中无条件触发的状态/存储更新——添加变更检测守卫，避免无变化时通知下游消费者。同时：如果包装函数接受 updater/reducer 回调，验证它是否尊重同引用返回（或"无变化"信号）——否则调用者的早期返回无操作会被静默击败
5. **不必要的存在性检查**：操作前先检查文件/资源是否存在（TOCTOU 反模式）——直接操作并处理错误
6. **内存**：无界数据结构、缺失清理、事件监听器泄漏
7. **过于宽泛的操作**：只需部分时读取整个文件、只需一个时加载全部项目

## 阶段三：修复问题

等待三个代理全部完成。汇总它们的发现并直接修复每个问题。如果某项发现是误报或不值得处理，记录并跳过——不争论发现本身，直接跳过。

完成后，简要总结修复了什么（或确认代码已经足够干净）。
